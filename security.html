Security & Privacy

Security by design,
not by default.

Every system built here treats security and data privacy as first-order requirements, not afterthoughts. This applies regardless of industry, scale, or budget.

Architecture Principles

How security is applied

These are not optional add-ons. They are the baseline on every engagement.

Encryption in Transit and at Rest

All data is encrypted in transit using TLS 1.2 or higher. Sensitive data at rest is encrypted at the field or column level where appropriate, using keys managed separately from the data store. This means that a breach of the database alone does not expose readable client data.

Role-Based Access Control

Access to data and functionality is governed by role — not by convention. Every system is built with a defined access model from the start: who can read what, who can write what, and what requires an audit event. Least-privilege principles are applied throughout.

Audit Logging and Event Trails

All significant system events — data reads, writes, modifications, exports, and access attempts — are logged in a tamper-resistant audit trail. Logs are structured, timestamped, attributed to a user or system actor, and retained according to your compliance requirements.

Secrets Management

Application secrets, API keys, database credentials, and encryption keys are never hardcoded or stored in source code. Dedicated secrets management services (Azure Key Vault, AWS Secrets Manager, or equivalent) are used with environment-scoped access and rotation policies.

Privacy-Compliant Architecture

Systems are designed with awareness of applicable privacy frameworks: the Australian Privacy Act, GDPR where relevant, and sector-specific regulations. Data minimisation, purpose limitation, and retention controls are built in — not added later as compliance theatre.

No Third-Party Data Sharing

Client data is not shared with third-party systems unless explicitly required by the engagement and approved in writing. AI models used in your system do not train on your data. No aggregation, no telemetry, no usage analytics that you have not authorised.

What you can expect

Ownership and assurances, by default.

These commitments apply to every engagement, without needing to be negotiated individually.

✓

Full code ownership

All source code is yours. There is no proprietary framework, no licence dependency, no lock-in. You can deploy it, modify it, or hand it to another engineer at any time.

✓

Infrastructure in your accounts

Systems are deployed to your cloud accounts, not shared infrastructure. You control access. You pay the cloud provider directly. There is no intermediary holding your environment.

✓

NDAs as standard

A mutual NDA is offered on all engagements as a baseline, before any scoping conversation takes place. No questions asked.

✓

No data retention after engagement

Post-engagement, copies of your data, schemas, or production credentials are not retained unless you explicitly request ongoing support with a defined access scope.

✓

Transparent dependencies

Every third-party service, library, or API used in your system is documented. You know exactly what your system depends on, what data those dependencies receive, and how to replace them if needed.

✓

Penetration testing support

For regulated environments, documentation and test access required for third-party security assessment can be provided. Architecture documentation suitable for legal and IT review is available on request.

For procurement and legal teams

A note on formal evaluation

If you are evaluating this engagement as part of a formal procurement process, vendor security questionnaires and technical architecture documentation for legal and IT review can be provided. If there are specific compliance frameworks your organisation must satisfy — ISO 27001, SOC 2, Australian Government security classifications, IRAP, or sector-specific standards — include these requirements in your enquiry and applicability will be assessed during scoping.

Engagements with regulated financial services clients, government contractors, and organisations subject to mandatory data breach notification obligations have been completed. The relevant obligations and design implications are understood.
